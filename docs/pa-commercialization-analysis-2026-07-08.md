## 2. 竞品/参考分析

### 2.1 Obsidian 官方

| 维度 | 内容 |
|---|---|
| **收费对象** | 服务（Sync $4-8/月、Publish $8/月） |
| **免费层** | 全功能本体，个人使用完全免费 |
| **商用 License** | 2026 年已改为可选，不再强制 |
| **付费逻辑** | 用户付的是"不用自己折腾同步/发布"的便利性 |
| **开源策略** | 本体闭源，插件 API 开放 |
| **参考价值** | Obsidian 自身选择了不锁功能、卖服务的路线 |
| **局限性** | 平台方和第三方插件面对的信任环境不同，不能直接类比 |

### 2.2 Obsidian Copilot

| 维度 | 内容 |
|---|---|
| **产品类型** | 第三方 AI 插件 |
| **定价** | Plus $139.99/年（托管模型，标注 Most Popular）；Supporter $349.99 一次性（BYOK 全功能 + 2 年 Plus） |
| **免费层** | $0 BYOK 基础功能 |
| **付费逻辑** | Plus 卖的是"免折腾"的便利性（托管模型）；Supporter 卖的是"BYOK 全功能永久授权 + 2 年托管" |
| **参考价值** | 唯一可参考的同类 AI 插件定价数据点 |
| **局限性** | 不知道其付费用户数、转化率或流失率。"Most Popular" 标签可能反映的是价格和门槛差异（Plus 便宜+方便），不一定是用户对两种付费哲学的选择。Copilot 有团队和后端基础设施，独立开发者的可行性不同 |

### 2.3 PKMer 社区

#### 它是什么

PKMer（Personal Knowledge Management + "er"）是中文 Obsidian 社区，定位"专注个人知识管理的人的聚集地"。不是单一插件，是**社区 + 产品矩阵**。

#### 收入来源

**（1）PKMer Market 插件 — 基础设施付费**

解决国内无法访问 Obsidian 官方插件市场的问题，提供 CDN 加速下载。

- 免费用户：100 次/月下载
- 会员：300 次/月下载
- 获取会员方式：直接付费 **或** 社区贡献（写教程、翻译、测试等）
- 收费逻辑：解决基础设施痛点（国内网络），不是锁功能

**（2）Thino（原 Obsidian Memos）— 产品付费**

PKMer 自研闪念笔记插件。

- Thino Pro：通过 PKMer 购买激活，早鸟价 ¥19.9 起，逐步提价（最终价格不详）
- 付费 = "内测优先权"——先用到新功能
- 公测版免费，但只有稳定性更新，不带新特性
- "延迟开源"：Pro 新功能最终会开源到公测版，时间不确定
- 官方表述："这个收费更类似于 Obsidian Insider 的逻辑"
- 保留社区贡献者免费获取通道——优秀贡献者直接赠送 Pro

**（3）Thino Web（规划中）**

- 独立 Web 版本，定价未公布
- 早期 Pro 购买者免费试用到正式定价出台
- 可能与 PKMer 会员打包

**（4）其他产品**

Simple MindMap（思维导图插件）、挂件集市、ZK 卡实例库——未见明确收费信息。

#### 社区积分体系（PKMer 模式的核心特色）

- 四类成员：普通用户 → 贡献者 → 管理者 → 捐助者
- 贡献（教程、翻译、测试、运营）→ PKMer 积分 → 兑换付费服务或升级权限
- 原则：多劳多得，承诺未来盈利后回馈贡献者
- 优秀贡献者/管理者享受所有付费项目折扣甚至免费
- 本质：用社区劳动力替代部分市场推广成本。贡献者写教程 = 免费内容营销

#### PKMer 模式对 PA 的参考价值

**事实层面的启示：**

- PKMer 证明在中文 Obsidian 生态，"社区先行 + 产品跟上"是走通了的路
- 先用内容（教程、文档、翻译）聚集用户，再用产品变现
- Thino 从 ¥19.9 起步逐步提价 = 极低门槛起步、社区口碑驱动价格上涨

**需要判断的差异：**

- PKMer 高度依赖中文社区运营能力（教程、QQ/微信群、B 站视频）。PA 定位出海 C 端，目标用户群和运营方式不同，社区路径不一定能复制到英文市场
- Thino 的"Insider 逻辑"（付费=早用新功能，最终免费）前提是迭代速度够快。迭代慢则付费用户觉得不值
- PKMer Market 的基础设施收费（CDN 加速）对 PA 无直接参考价值——PA 没有类似的基础设施痛点可变现

**无法判断的：**

- PKMer 实际的付费用户数和收入规模。搜索结果未披露，无法评估商业成功程度
- Thino 的"逐步提价"最终定在什么价格。¥19.9 和 Copilot 的 $139.99/年 之间差距巨大，中间的实际接受度是未知的

## 3. 可以确定的结论

以下判断有代码证据或逻辑支撑，置信度较高：

### 3.1 Tier gate 工程准备充分

gate 已在代码里，以"不改变现有行为"的方式实现（MOCK_LICENSE_TIER = "paid"）。未来启用收费不需要重新设计架构，只需把 mock 换成真实 license provider。这比"以后再加 gate"好得多——后者往往导致免费用户感知到功能被拿走。

### 3.2 AGPL-3.0 保护到位

License 决策已完成，不需要再讨论。

### 3.3 当前瓶颈是用户量，不是收费机制

产品没有外部用户信号（复盘确认：所有产品决策来自内部循环）。不管选择什么定价模型，第一步都是让人知道并开始用。社区透出计划是当前最高优先级的商业化准备动作。

### 3.4 paid tier 的锁定逻辑需要澄清

5 个 paid capability 中多数涉及 AI 调用。但在 BYOK 模式下用户自付 API 费用，锁 AI 能力的合理性需要重新审视：

- **锁成本** → 适合托管模式（开发者承担 API 费用，用户付 subscription）
- **锁价值** → 适合 BYOK 模式（用户自付 API 费用，付的是功能授权费）

两种逻辑导向不同的定价故事，目前代码里混在一起。

## 4. 三条可能的收费路径

| 路径 | 前提 | 额外工程量 | 风险 |
|---|---|---|---|
| **A. BYOK + license key 买断/年付** | 有人愿意为功能价值付费 | license 校验（本地 key + 轻量远程验证） | BYOK 用户已付 API 费，付费意愿未知 |
| **B. 托管 API + subscription** | 有人想要"不折腾 API key"的便利性 | 后端基础设施（API 路由、用量计费、用户管理） | 工程量大，运维负担，独立开发者资源紧张 |
| **C. 全免费 + 生态变现** | 插件的价值是引流/建立品牌 | 其他产品线的变现设计 | 插件本身不产生收入 |

**Copilot 参考**: 其 Plus（托管）标注 "Most Popular" 暗示路径 B 在它的用户群里可行，但不能证明 A 不可行或 C 不值得。且 Copilot 有团队和后端——同路径对独立开发者可行性不同。

**PKMer 参考**: 走的是路径 A 的变体（付费 = 内测优先权 + 延迟开源），加上社区积分体系降低了用户对"付费"的抵触。但 PKMer 是社区驱动模式，需要持续的社区运营投入。

## 5. 需要决策的开放问题

### Q1: 插件本身的收入定位

> 这个插件是用来赚钱的，还是用来建立影响力然后在其他产品线赚钱的？

- 前者 → 需要尽早设计收费路径（路径 A 或 B）
- 后者 → 全免费 + 最大化传播（路径 C）

### Q2: 托管层的投入意愿

> 愿意投入多少精力在后端基础设施上？

- "不愿意"或"很少" → 托管层不可行，只能围绕 BYOK 设计（路径 A 或 C）
- "愿意投入" → 路径 B 成为选项，但这是一个显著的工程方向切换

### Q3: MOCK_LICENSE_TIER 的切换时机

> 什么时候把 mock 换成真实校验？

- **太早切** → 在没有用户时限制产品完整体验，对获客不利
- **太晚切** → 用户把所有功能当成免费并形成预期，再加 gate 产生反感
- **PKMer 的做法参考**: 先全免费/低价积累用户，逐步提价。但 PKMer 的"延迟开源"模式（Pro 先用，公测跟上）提供了一种中间方案——付费买的是时间差而非功能差

### Q4: 社区透出时是否暴露 paid tier

> 社区透出的时候就展示"有付费层"，还是先全功能开放再引入收费？

- **展示** → 早期用户建立了"有付费层"的预期，后续切换摩擦小
- **不展示** → 降低早期用户决策门槛

### Q5: paid capability 的选择标准

> 当前 5 个 paid capability 的选择逻辑是什么？如果采用 BYOK 路径，需要重新评估哪些能力值得 gate。

## 6. 分析过程中的教训

### 6.1 Memory 与代码实际状态不一致

Memory 记录 License 为 "Apache 2.0 考虑切换到 GPL-3.0 或 BSL 1.1"，但代码已经是 AGPL-3.0-only。分析中引用了过期的 memory 信息导致错误结论。

**原则**: 涉及代码状态的判断必须验证代码，不能依赖 memory。

### 6.2 避免将推断表述为结论

早期分析中出现 "Obsidian 社区对'免费核心 + 付费便利'模式的接受度远高于'免费试用 + 付费解锁'" 的断言。拆解后发现：

- 依据仅有 Obsidian 自身定价和 Copilot 一个数据点
- 两个数据点 → "远高于"是过度推断
- 更诚实的表述是："Obsidian 的平台商业模式和 Copilot 的定价结构暗示'付费便利'在这个生态里有可行性，但目前没有直接证据比较两种模式的接受度"

**原则**: 区分"事实"、"合理推断"、"无法判断"三个层次，不把推断当结论。
