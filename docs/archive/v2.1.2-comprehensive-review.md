### 2.2 架构与过度设计

> 上下文：`docs/pa-agent-architecture-plan.md` 是当前架构权威。已定决策（`[[project_action_mode_roadmap]]`）：三层工具管线 / PolicyEngine / 双事件系统是为 action mode 预留的"过早抽象"。本轮去**实测**这些抽象是否对得起复杂度，而非重新论证。

#### A-1. 三层 `ToolRegistry → CoreToolProvider → CapabilityRegistry`（最大冗余）

- 位置：`chat-tool-registry.ts` 451L + `core-tool-provider.ts` 75L + `capability-registry.ts` 284L + `capability-adapter.ts` 197L
- 观察：两套 registry 表面方法几乎完全相同（`getDefinition`/`listDefinitions`/`execute`/`prepareAndValidate`/`exportProviderSchemasSafe`/`has`）。`CoreToolProvider` 实例化一个 `ToolRegistry`，注册 9 个核心工具，然后 `loadCapabilities()` 把每个工具用 lambda 适配成 `AgentCapability`，再注册进 `CapabilityRegistry`；`CapabilityRegistry.execute → capability.execute → legacyRegistry.execute`。`ChatToolRegistryDefinition` 在三处被 clone
- 判断：**真冗余**（CapabilityRegistry 已经是 action mode 入口，legacy ToolRegistry 没有独立调用方）
- 动作：保留 `CapabilityRegistry`，把 `assertObsidianOperationsV1AToolPolicy` + 输出预算迁进去；改 `chat-tool-factories.ts` 直接生产 `AgentCapability`，删 `ToolRegistry`、`core-tool-provider.ts`、大半 `capability-adapter.ts`
- **预估 −500 LOC**

#### A-2. 双事件流 + 100 行 LegacyEvent adapter

- 位置：`agent-runtime-primitives.ts:114-166`（AgentEventEmitter）+ `chat-types.ts:388-432`（LegacyAgentEvent*）+ `pa-agent-runtime.ts:355-460`（CanonicalToLegacyEventAdapter）
- 观察：v2 `AgentEvent` → 适配器逐事件转成 v1 `LegacyAgentEvent`。唯一外部消费者是 `chat-view.ts:2131 onLifecycleEvent` 已收 v2，加上同一文件用 v1 渠道收 `tool-running/tool-done` 状态映射。v2 信息已足
- 判断：**真冗余**（不影响 action mode）
- 动作：chat-view 状态映射改读 v2 AgentEvent，删 LegacyAgentEvent 系列 + Emitter + adapter
- **预估 −220 LOC**

#### A-3. `PolicyEngine` 是伪 engine

- 位置：`policy-engine.ts:15-52`
- 观察：4 个 public 方法全部 → 同一 `evaluate()`，5 个静态分支；v1 read-only 下每次都返回 `allowed: true`；只被 `CapabilityRegistry.register/execute` 调用
- 判断：为 action mode 预留 ✓，但**当前 API 形态过度**
- 动作：塌缩为 `canUseCapability(cap, platform)` 一个纯函数放 `capability-types.ts`；action mode 真要状态再升级回类
- **预估 −40 LOC + 一层注入**

#### A-4. `chat-tool-*` 8 文件按动词切分非分层

- 位置：execution-helpers 1346L / factories 831L / guards 280L / prepare-helpers 173L / constants 63L / types 342L / registry 451L
- 观察：`execution-helpers` 仅被 factories/registry/guards 内部消费；guards 同时 import constants+execution-helpers+types；factories import 全部 4 个 sibling。是 hub-spoke 星型，无层级
- 判断：**部分真冗余**（不属于 action mode 预留）
- 动作：把 constants/prepare-helpers 合入 types；guards 合入 registry；execution-helpers 的 per-tool 序列化函数搬回 factories 对应工厂旁。文件 8→4
- **预估 −100 LOC 样板**

#### A-5. `pa-agent-runtime.ts` 巨石：4 类 + 600 LOC 自由函数

- 位置：`pa-agent-runtime.ts:355-1584`
- 观察：内联 `CanonicalToLegacyEventAdapter`(~100) + `ChatPlanner`(~55) + `MemorySearchTool`(~135) + 600 行 nativeToolCall 解析/schema 过滤/prompt 拼装。`streamPaAgentCanonicalTurn`（497-621）大量 `const x = this.x` 闭包捕获，因为内联 async generator 不能用 `this`
- 判断：**结构冗余**，非 action mode 预留
- 动作：拆出 `chat-planner.ts` / `memory-search-tool.ts` / `native-tool-parsing.ts` / `canonical-turn-runner.ts`；把 model builder 提为顶层函数显式传参
- **预估 −50 LOC 净，可读性大幅改善**

#### A-6. `AgentMessageUpdate` 9 变体过细

- 位置：`chat-types.ts:251-260`
- 观察：`thinking_start/_delta/_end` × `text` × `toolcall` 共 9 变体，但只有 `thinking_delta` 实际被消费；其余 `_start/_end` 仅触发空 case
- 判断：**真冗余**
- 动作：合并为 `{ kind: "delta"; phase; text }` + `{ kind: "phase"; phase; transition }`
- **预估 −80 LOC**（含 chat-view reducer）

#### A-7. `ToolExecutionOutcome` 8 分支多数被压平

- 位置：`chat-types.ts:262-270`；消费在 `pa-agent-answer-completion-policy.ts:245-257`
- 观察：completion policy 已把 recoverable_error / schema_invalid / policy_rejected / budget_exceeded 收成同一 "failure" 桶；`abort_timeout` 几乎只在 loop 自身分支
- 判断：**真冗余**
- 动作：改为 `success | failure | aborted` + `failureReason` 字段
- **预估 −40 LOC switch**

#### A-8. `pa-agent-host-tools.ts` 与 `core-tool-provider.ts` 边界

- 位置：两文件
- 观察：host-tools 411L 完全是 PA loop ↔ CapabilityRegistry 桥；core-tool-provider 是注册侧桥。它们是同一缝合面的两端
- 判断：host-tools **设计正确**（PA loop 契约的实现）；core-tool-provider 的多余性已在 A-1 处理

**总体结论**：13,313 LOC 的 `ai-services/` 中，剔除 action mode 故意预留后，**真正可以现在删/合并的净代码量估计 −900 至 −1,100 LOC（约 7–8%）**，最大三块来自 A-1（registry 三层包装）、A-2（双事件流）、A-5（内联巨石的横向再切分）。

### 2.3 性能与包体积

> 关键数据：`dist/main.js` = 3.22 MB（gzip 1.12 MB），其中 WASM base64 = 1.22 MB 占 38%；剩余 ~2 MB 中 LangChain + openai SDK + langsmith + js-tiktoken 占大头。esbuild `splitting:false`、`format:cjs`——所有 `import()` 仍被内联到单 bundle。

#### Perf-1. SQLite WASM 拆成独立 .wasm asset 而非 base64 内联 ⭐ 本轮最大单点

- 位置：`esbuild.config.mjs:78-119`（`lazyBinaryPlugin`）、`src/vss/sqlite-inline-assets.ts:2`
- 当前：1,285,140 字节 base64 字符串塞进 main.js，V8 仍需 lex 这 1.22 MB 文本（移动端 ~80-120 ms 解析税）。decode 已懒，但 string literal 本身常驻
- 改法：esbuild plugin 改为 `copy` 到 `dist/sqlite3.wasm`，运行时 `fetch(this.app.vault.adapter.getResourcePath(...))` 或 base64 落盘到 plugin dir；release zip 多带一个文件
- 预估收益：**main.js 3.22 MB → 2.0 MB（-38%）**，gzip 1.12 MB → ~0.8 MB，冷启动解析节省 ~80 ms
- **本轮决策：A3 不动**（见 §3.1）

#### Perf-2. LangChain + openai SDK 强绑定在启动路径

- 位置：`src/plugin.ts:6` → `chat/chat-view.ts:2` → `ai-services/chat-service.ts` → `ai-utils.ts:3-4`（`ChatOpenAI`、`OpenAIEmbeddings`）；`@langchain/openai` 经 `require("openai")` 链拉入完整 openai SDK + js-tiktoken + zod + langsmith
- 当前：所有 LangChain 代码在 plugin onload 即解析完成，估算 ~600-900 KB 在 main.js
- 改法：`plugin.ts` 只 `import type` chat-view；`registerView` 工厂里用 `await import("./chat/chat-view")`；把 `ai-utils.ts` 内的 LangChain 实例化挪到 `AssistantHelper.generate()` / `ChatService.send()` 首次调用前的 `await import("@langchain/openai")`
- 预估收益：冷启动解析省 ~150 ms（mid 移动端），main.js parse-time 显著下降

#### Perf-3. `SettingTab` 在字段初始化器中 eager 构造

- 位置：`src/plugin.ts:109` `private settingTab: SettingTab = new SettingTab(this.app, this);`、`src/settings.ts:4` 顶层 `import Picker from "vanilla-picker"`
- 当前：onload 之前就跑 settings.ts 全部 1985 行 + vanilla-picker（~30 KB 打包后）
- 改法：字段改为 `undefined`，在 `addSettingTab` 调用时 `new SettingTab(...)`；`Picker` 改为 `display()` 里 `await import("vanilla-picker")`
- 预估收益：~40 KB + 一次性解析省 ~10 ms；解锁 Picker 真正按需

#### Perf-4. chat-view 3314 行整体 eager + 每次 onOpen 重建 DOM

- 位置：`src/chat/chat-view.ts:193-2520`（onOpen），`src/plugin.ts:6` eager import
- 当前：onOpen 同步创建 ~40 个 DOM 节点（textarea/buttons/menus/composerHint…），每次开关都重建；mermaid 模块 line 13 eager
- 改法：(a) plugin.ts 改 `registerView` 用动态 import；(b) onOpen 把 composer chrome 缓存到 `this`，复用而非重建；(c) `./mermaid` 改成 `await import` 在 `containsMermaidFence` 命中时才拉
- 预估收益：冷启动 ~80 KB；每次开聊省 ~10-20 ms DOM 重建

#### Perf-5. `registerView(STAT_PREVIEW_TYPE/RECORD_PREVIEW_TYPE)` 把 React + react-chartjs 带入冷路径

- 位置：`src/plugin.ts:191-204`、`src/stats-view.ts:2-3` `import { createRoot } from "react-dom/client"`、`src/preview.ts:4-5`
- 当前：React 18 (~140 KB minified) 在 onload 解析完成。Chart.js 已经在 `Statistics.tsx:23` 用 `lazy()`，但 `splitting:false` 时 esbuild 把整个 Chart.js（~250 KB）依旧内联进 main.js —— `lazy()` 只是延后 React render，不省 parse
- 改法：`registerView` 工厂内 `const { Stat } = await import('./stats-view')`；stats-view.ts 内部再动态 `import('react-dom/client')`
- 预估收益：~150 KB 冷路径解析，~30 ms

#### Perf-6. `memoryManager.startAutoMaintenance()` 紧跟在 onload 主线程

- 位置：`src/plugin.ts:206-208`
- 当前：onload 同步路径上又跑 `await this.updateMemoryStatusBar()`，阻塞 `onLayoutReady` 之前
- 改法：用 `this.app.workspace.onLayoutReady(() => { ... })` 包裹，已对 calloutManager 这么做了
- 预估收益：感知冷启动 -20~50 ms（取决于 status bar 渲染）

#### Perf-7. `monkeyPatchConsole` import 路径常驻

- 位置：`src/plugin.ts:17`，只在 `settings.debug=true` 才用
- 改法：`if (this.settings.debug) { const { monkeyPatchConsole } = await import('./obsidian-hack/...'); }`
- 预估收益：~2-3 KB（可忽略），但减少 import graph 噪声

#### Perf-8. CSS 52 KB —— Tailwind preamble 重复

- 位置：`styles.css:1` 起 `*,:after,:before{--tw-…}` 大段 CSS var 在 `::backdrop` 等选择器中重复。`tailwind.config.cjs` content glob 范围合理（src/**），但未跑 `cssnano`

**总评**：v2.1.2 启动路径**最大的一颗钉子是 SQLite WASM 的 1.22 MB base64 内联在 main.js**（Perf-1）——它单独吃掉 bundle 的 38%、是 V8 解析时间的最大单一来源，且即使用户从不开 Memory 也得付完整 lex 成本。lazyBinaryPlugin 已解决 decode/heap，下一步必须把字节挪出 main.js。**本轮决策接受不动**（见 §3.1）。

### 3.5 ⑤ @sqliteai/sqlite-wasm 供应商脱钩 — **决策：v2.3 进 roadmap**

#### "供应商风险"的具体含义

`@sqliteai/sqlite-wasm` 不是 SQLite 官方维护，是 SQLiteAI 公司的商业产品的开源副产品：

1. **维护节奏不可控**：当前版本 `3.50.4-sync.0.8.30-vector.0.9.23` 发布 2025-09，8 个月没更新；upstream 已发 `...1.0.0` 系列（项目还活，只是**你的版本卡住了**）
2. **复合版本号是它自己的约定**：`sqlite版本+sync版本+vector版本` 拼起来，社区其他人遇到同样问题的概率极低，**Stack Overflow 搜不到**
3. **用户基数小**：npm weekly downloads 估计三位数（对比 `better-sqlite3` 月下载几十万），**bug 大概率只能等官方修**
4. **迁出成本未知**：若供应商 archive 仓库或出严重 bug，需要在短时间内迁到 `sql.js + sqlite-vec`，**工作量没人帮你估过**

**核心问题不是"现在出事了"，而是"6 个月后他们停更，你能多快迁出去"**

#### 调研发现：实际"私有依赖"极薄

`src/vss/sqlite-worker.ts` 对 @sqliteai 的真实绑定仅 **3 个非标准 SQL 函数**：

| 函数 | 用途 | 调用位置 |
|---|---|---|
| `vector_init('table','col','type=FLOAT32,dimension=1024,distance=COSINE')` | 注册向量列 | `:408` |
| `vector_as_f32(?)` | Float32Array → BLOB | `:440, 553, 600` |
| `vector_full_scan(table, col, vec, k) → (rowid, distance)` | KNN top-k（**brute-force**，函数名直接暴露） | `:553, 600` |

其他全部用上游 SQLite 标准 API：
- DDL/DML：`CREATE TABLE/INDEX`、`INSERT/UPDATE/DELETE`、`INSERT OR REPLACE`、JOIN、子查询、`ORDER BY/LIMIT`、prepared statement bind/step/finalize
- 事务：`BEGIN/COMMIT/ROLLBACK`
- PRAGMA：仅 `page_size/page_count`
- FTS5 contentless 表 + `unicode61 remove_diacritics 2` tokenize + `MATCH`、`ORDER BY rank`、`integrity-check`
- 中文分词在 JS 层用 `Intl.Segmenter`（`fts-query-builder.ts:7-15`），把 CJK 转 phrase query 喂给 unicode61
- OPFS-SAH 持久化：`installOpfsSAHPoolVfs`、`OpfsSAHPoolDb`（**sqlite.org 上游 API**，非 @sqliteai 私有）

**关键洞察**：@sqliteai 给你的"vector 加速"实际上没有真加速——它没有 HNSW/IVF 索引，只是用 C 实现了 brute-force 全表扫。**JS 端 30 行 Float32 cosine 完全等价**。

#### 六个候选方案完整对比

| 方案 | 难度 | 主要障碍 | 估时 |
|---|---|---|---|
| **A. `sql.js`** | 大 | 无 OPFS-SAH 持久化（要自己 IndexedDB 持久化）；无 vector | 7-10 d |
| **B. `wa-sqlite`** | 中 | 需自编译进 sqlite-vec；OPFS-SAH 支持有但 API 不一样 | 5-7 d |
| **C. `@sqlite.org/sqlite-wasm`（采纳）** | 小 | API 完全兼容（@sqliteai 就是 fork 它）；FTS5/OPFS-SAH 现成；**唯一缺 vector 函数** | 1 d 替换 + 向量层重写 |
| **D. `sqlite-vec` 扩展** | 大 | upstream @sqlite.org/sqlite-wasm 不支持运行时加载扩展，要自编译 WASM；vec0 virtual table API 不同 → 三处 SQL 全改 | 7-10 d |
| **E. 纯 JS（hnswlib-wasm/usearch + minisearch + 自持久化）** | 大 | 抛掉整个 SQLite 层；要重写 schema/事务/FTS 中文分词集成/持久化（OPFS or IndexedDB）/统计；移动端 hnswlib-wasm 内存表现待验 | 8-12 d |
| **F. `duckdb-wasm`** | 大 | bundle 30 MB+，移动端杀伤太大；与现有"WASM 内存优化"卖点冲突 | 不建议 |

#### 最务实方案：C + JS brute-force 化

**步骤**：

1. `package.json`：`@sqliteai/sqlite-wasm` → `@sqlite.org/sqlite-wasm`（SQLite 官方维护的 WASM 发行）
2. `src/vss/sqlite-worker.ts` 3 处 `vector_*` SQL 改为：`SELECT id, embedding FROM vss_chunks` 一次性拉出全部 Float32 BLOB → JS 端 cosine/L2 求 top-k
3. Worker 内做热向量 cache：首次查询全表 → 内存 Float32Array → 后续查询零 SQL 开销
4. RRF/FTS/事务/schema/Worker 协议 **零改动**
5. 跑 `__tests__/vss-*.test.ts` 全套回归 + 真机 vault 10k chunk 性能 sanity

**性能预算**：

- 典型 Obsidian vault < 10k chunk
- 10k × 1024 dim × 4 bytes = **40 MB** 一次性常驻 worker
- Float32Array cosine top-k 在 worker 里几毫秒（不阻塞主线程）
- 与现状 `vector_full_scan` 复杂度持平

**硬收益对比**：

| | 现在 | 迁移后 |
|---|---|---|
| 供应商 | @sqliteai 单一商业 fork | SQLite 官方 + 你的 30 行 JS |
| WASM 来源 | 8 个月没更新的 fork | 官方维护、Stack Overflow 找得到 |
| Bundle | 1.22 MB（含 sqlite-vector + sqlite-sync 编译产物） | **可能更小**（少了 vector/sync 扩展） |
| 风险 | 6 个月后供应商挂了你 3-5 天迁不出去 | 永久脱钩 |
| 性能 | brute-force | brute-force（同等） |

**预估工作量**：3-5 天（含 worker 内热向量缓存 + 测试回归 + 真机验证）

#### 过渡期策略（v2.2）

- CI 加一个月度 spike 跑 `npm install @sqliteai/sqlite-wasm@latest` 全量回归（E2 策略）
- upstream 有重要修复就跟

#### Plan B 不写代码原则

纯 JS 向量库（hnswlib-wasm / usearch）或 wa-sqlite + 自编译 sqlite-vec 是次优选项，工作量 7-10 天。

**仅在 @sqlite.org/sqlite-wasm 出现阻塞性问题时启动**。

不写"防御性代码"——避免为可能永远不发生的风险写真实需要维护的迁移代码（典型过度工程陷阱）。

## 6. 决策原则总结（为后续 review 留 baseline）

本轮 review 形成 / 强化的 3 条判断逻辑，建议成为后续 release 评审的默认 baseline：

### 6.1 P0 产品硬伤优先于内部架构精简

8 条 P0（首单/留存/文档/Notice）总耗时 < 2 天，对新用户首单的杠杆 **远高于** 内部 −1,080 LOC 的代码清理。

后续 review 默认顺序：**产品/UX → 性能 → 架构精简**

### 6.2 "Plan B 不写代码"反过度工程原则

调研路径要清晰（如 §3.5 @sqliteai 六候选完整对比），但 plan B 代码不要真写。

理由：未发生的风险写代码 = 真实维护成本 + 大概率永远用不上。

把 plan B **写进 memory** 而非写进代码，是更轻量的"组织记忆"形式。

### 6.3 "复议触发条件"代替"无条件保留"

对于"现在 ROI 不够但未来可能值"的优化（如 React → Preact），不直接拒绝、不无条件保留，而是定义**触发条件**写进 memory（如 `components ≥ 5 时复议`）。

避免 review 决策衰减——下次 review 时不需要重新论证，触发即评估。

### 7.3 名词解释

- **WASM**: WebAssembly，浏览器 / Electron 中跑 native 编译产物的二进制格式。SQLite WASM = SQLite 编译成 WASM 在 Obsidian 内跑。
- **base64 内联**: 把二进制资产编码为 base64 字符串塞进 JS 源码，避免额外文件加载请求；代价是 JS 体积膨胀 ~33%、V8 lex 文本成本。
- **OPFS-SAH**: Origin Private File System Synchronous Access Handle，浏览器 / Electron 提供的私有文件系统同步访问接口，SQLite WASM 用它做持久化。
- **FTS5**: SQLite 的全文检索扩展（version 5），支持 tokenizer、phrase query、`MATCH`、`ORDER BY rank`。
- **RRF**: Reciprocal Rank Fusion，把多路检索结果（如 vector + FTS）按倒数排名分数融合。本项目在 JS 层实现，与 SQLite 解耦。
- **HNSW / IVF**: 两种主流向量索引算法。HNSW = Hierarchical Navigable Small World，近似 KNN；IVF = Inverted File Index。本项目当前**不用任何加速索引**，是 brute-force 全表扫。
- **brute-force 全表扫**: 不建索引，每次查询都遍历全部向量计算距离。在 < 10k 向量规模上 JS 端几毫秒即可完成。
- **capability kinds**: PA Agent 架构中的能力分类抽象——tool（可执行工具）/ context（提供上下文）/ action（写操作，v1 不暴露）。三类是 action mode roadmap 的核心契约，**不可合并**。
- **action mode**: PA Agent 未来的写操作模式（v1 仅 read-only + network-read），需要 preview/confirmation/rollback/audit/permission gate 完整设计。三层 ToolRegistry / PolicyEngine / 双事件系统是为此预留的"过早抽象"。
- **Preact alias**: 通过打包器配置把 `import 'react'` 重定向到 `preact/compat`，业务代码不改、运行时换成轻量 Preact。
- **keychain 迁移**: 把明文存储的 API token 迁到 OS keychain（macOS Keychain / Windows Credential Manager / Linux Secret Service）。本项目已部分实现，apiToken 明文链待 v2.5 删除。

**核心原则**：bundle size 不作为产品决策驱动力。Obsidian 用户群对插件几 MB 差距无感知。

**受影响的决策与理由调整**：

| 决策 | 原 review 文档理由 | 修正后理由 |
|---|---|---|
| ① WASM 不动 (A3) | "拆包能省 38% bundle 但工作量大" | **无用户痛点驱动**：移动端无冷启动/OOM 反馈，bundle 大小不再是评估维度 |
| ③ React → Preact 不切 (C1) | "100KB 收益对比 1.22MB WASM 不成比例" | **无用户痛点驱动**：bundle 不是问题，React 当前可用，无需折腾 |
