### A.3 `bruteForceTopK()` Algorithm Design

```typescript
// src/vss/brute-force-search.ts

/**
 * Brute-force cosine similarity top-k search over a vector cache.
 *
 * Performance: P50 ~14ms for 10k vectors x 1024 dims (from spike report).
 * The function is allocation-light: it reuses a fixed-size min-heap
 * instead of sorting the full result set.
 */
export function bruteForceTopK(
    queryVec: Float32Array,
    cache: Map<number, Float32Array>,
    k: number,
    metric: "COSINE" | "L2" = "COSINE",
): { id: number; distance: number }[] {
    // Min-heap of size k (by distance ascending = worst match at top)
    const heap: { id: number; distance: number }[] = [];

    for (const [id, vec] of cache) {
        const distance = metric === "COSINE"
            ? cosineDistance(queryVec, vec)
            : l2Distance(queryVec, vec);

        if (heap.length < k) {
            heap.push({ id, distance });
            if (heap.length === k) heapify(heap);
        } else if (distance < heap[0].distance) {
            heap[0] = { id, distance };
            siftDown(heap, 0);
        }
    }

    return heap.sort((a, b) => a.distance - b.distance);
}

function cosineDistance(a: Float32Array, b: Float32Array): number {
    let dot = 0, normA = 0, normB = 0;
    for (let i = 0; i < a.length; i++) {
        dot += a[i] * b[i];
        normA += a[i] * a[i];
        normB += b[i] * b[i];
    }
    const denom = Math.sqrt(normA) * Math.sqrt(normB);
    if (denom === 0) return 1; // max distance for zero vectors
    return 1 - dot / denom;
}

function l2Distance(a: Float32Array, b: Float32Array): number {
    let sum = 0;
    for (let i = 0; i < a.length; i++) {
        const d = a[i] - b[i];
        sum += d * d;
    }
    return Math.sqrt(sum);
}

// Max-heap helpers (heap[0] = largest distance = worst match)
function heapify(heap: { distance: number }[]): void { /* standard buildHeap */ }
function siftDown(heap: { distance: number }[], i: number): void { /* standard siftDown */ }
```

**Key design decisions:**

1. **Min-heap of size k** instead of full sort: reduces time complexity from O(n log n) to O(n log k). For n=10k and k=10, this saves ~30% vs the naive sort approach noted in the spike report.

2. **`Float32Array` throughout**: no boxing to `number[]`. The cache and query vector are both `Float32Array` to avoid GC pressure and enable V8's SIMD autovectorization.

3. **Pure function, no state**: the cache is injected. This makes the function trivially testable in Jest without Worker scaffolding.

4. **Both COSINE and L2**: the existing `EmbeddingProfile` (line 23 of `src/vss/types.ts`) declares `distanceMetric: "COSINE" | "L2"`, so the brute-force must support both.

### A.4 Hot Vector Cache Design

Implemented inside `sqlite-worker.ts` as worker-scoped state (not exported):

```typescript
// Worker-scoped cache
let vectorCache: Map<number, Float32Array> | null = null;

async function getOrLoadVectorCache(): Promise<Map<number, Float32Array>> {
    if (vectorCache !== null) return vectorCache;
    const rows: unknown[][] = [];
    requireDb().exec({
        sql: "SELECT id, embedding FROM vss_chunks",
        rowMode: "array",
        resultRows: rows,
    });
    const cache = new Map<number, Float32Array>();
    for (const row of rows) {
        const id = row[0] as number;
        const blob = row[1] as Uint8Array;
        cache.set(id, new Float32Array(
            blob.buffer, blob.byteOffset, blob.byteLength / 4
        ));
    }
    vectorCache = cache;
    return cache;
}
```

**Cache invalidation sites:**

| Mutation path | Invalidation strategy | Location in `sqlite-worker.ts` |
|--------------|----------------------|-------------------------------|
| `upsertFile()` | After COMMIT: add new chunk vectors to cache via `cache.set(id, vec)` for each inserted chunk. Read back IDs with `SELECT id FROM vss_chunks WHERE path = ?`. | Lines 418-481 |
| `deleteFile()` | Before SQL delete: collect chunk IDs via `SELECT id FROM vss_chunks WHERE path = ?`, then `cache.delete(id)` for each. | Lines 483-498 |
| `reset()` | Set `vectorCache = null`. Next query triggers full reload. | Lines 757-775 |
| `dispose()` | Set `vectorCache = null`. Worker memory freed. | Lines 777-787 |

**Memory budget:** 10k chunks x 1024 dims x 4 bytes = ~40 MB. The spike report confirms this is within iOS Safari worker limits. For extreme vaults (50k+ chunks), a chunked loading strategy is the fallback (not in v2.3 scope).
