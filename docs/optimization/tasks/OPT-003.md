# OPT-003 — Make PA Store persistence atomic and retry-safe

## Pre-implementation revalidation

- Original problem: Review Queue, Memory Governance, and Saved Insight mutate their live arrays before persistence. A rejected save leaves a ghost state; Memory retry creates duplicates. Plugin adapters also leave the failed next snapshot in `settings`.
- Runtime reproduction on the immutable baseline: all three calls threw `disk`, but snapshots reported one newly added item/record; Memory retry separately produced two IDs.
- Root cause: mutation is committed before persist, with no serialization or adapter rollback.
- Modification boundary: three Store implementations, their tests, PluginManager adapters, and plugin integration tests (8 files total).
- Non-goals: no data-format, state-machine, ID, dedup, public API, schema, authorization, or confirmation-flow change.
- Acceptance:
  1. A rejected create/update/lifecycle persist leaves Store snapshot exactly unchanged.
  2. A retry creates only one durable logical object.
  3. Concurrent mutations are persisted serially and do not allow an older snapshot to finish last.
  4. Plugin settings subtrees are restored when `saveSettings()` rejects.
