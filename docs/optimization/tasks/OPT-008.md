# OPT-008 — Add PR and master validation CI

## Pre-implementation revalidation

- Original problem: the only build workflow runs on release tags, so repository
  changes have no versioned PR or `master` test/lint/build gate before release.
- Root cause: release publication and change validation share one tag-only
  workflow; no read-only CI workflow exists.
- Confirmed user decision: validate both `pull_request` and pushes to `master`,
  cancel superseded runs, and use the same complete checks as release without
  release permissions or publication steps.
- Modification boundary: add `.github/workflows/ci.yml` only.
- Non-goals: no change to `release.yml`, branch-protection API mutation,
  dependency/action upgrade beyond versions already used by release, commit,
  push, or remote required-check configuration.
- Acceptance: minimal `contents: read`; Node version and npm cache from current
  project metadata; `npm ci`, notices, serialized coverage, lint, build, and
  bundle audit; valid YAML; no release/write step.

## Implementation

- Added one `CI / validate` job for PR and `master` triggers.
- Concurrency groups by PR number or ref and cancels superseded runs.
- Reused `actions/checkout@v6` and `actions/setup-node@v6` from the existing
  release workflow.
- Local Ruby YAML parsing and `git diff --check` pass. `actionlint` is not
  installed; the first GitHub Actions run remains the runner-level proof.
- Remote branch protection cannot be configured until the workflow is committed
  and pushed, and remains outside this run's no-commit/no-push boundary.

## Independent gates

- Review: ACCEPT; no blocking finding.
- Verification: PASS. Direct untracked-workflow inspection, Ruby YAML parse,
  full coverage Jest, TypeScript, lint, notices, build, bundle audit and
  whitespace checks passed.
- `actionlint`, first remote runner execution and required-check/branch
  protection remain external follow-up gates after commit/push.

## Rollback

- Delete only the uncommitted `.github/workflows/ci.yml` addition.
- Do not modify the existing tag-triggered release workflow.
- Because this run did not commit, push, or configure branch protection, there
  is no remote workflow/check/state to undo.
- Re-run YAML parsing and `git diff --check` after the inverse change.
