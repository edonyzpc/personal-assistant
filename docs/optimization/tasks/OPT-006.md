# OPT-006 — Record-first reversible automatic Memory

## Pre-implementation revalidation

- Original problem: Level 2 creates an active `ConfirmedMemoryRecord` and marks
  the originating Review Queue item `applied`, but the generated Memory ID is
  not linked back to the queue. Applied candidates are absent from the candidate
  payload, and the existing candidate Remove callback tries the invalid
  `applied -> dismissed` transition. The product therefore cannot display and
  safely remove the durable record it accepted automatically.
- Reproduced code path:
  `PluginManager.createReviewQueueItem()` ->
  `confirmMemoryCandidateQueueItem()` ->
  `MemoryGovernanceStore.confirmCandidate()` -> queue `applied`; the returned
  record ID is discarded. `MemoryGovernanceSection` renders records read-only.
- Confirmed user decisions:
  1. Reaching 30 confirmations silently enables Level 2; no one-time prompt.
  2. `ConfirmedMemoryRecord` is the canonical governance object.
  3. Active records are removable by exact record ID after confirmation; the
     original note is untouched and a text-free tombstone remains.
  4. New records retain their originating queue item ID so successful removal
     can move that audit item to `undone`; legacy records are never guessed or
     heuristically matched.
  5. Users can pause/resume automatic Memory in settings without resetting the
     monotonic trust count. While paused, new candidates remain manual.
- Modification boundary: Memory governance record schema/store, the existing
  candidate-confirm flow, Pagelet detail callback threading and record cards,
  the Memory settings toggle/default/normalization, bilingual locale keys, and
  direct regression tests.
- Non-goals: no retrieval weighting, no migration of historical suggested
  candidates, no fuzzy legacy linkage, no raw provider/source text persistence,
  no changes to Memory/VSS indexing, and no public API/schema/auth changes.
- Acceptance:
  1. Auto and manual confirmation records persist `originReviewQueueItemId`.
  2. Paused Level 2 creates a suggested queue item and no Memory record.
  3. Active records show source evidence; auto records show the automatic badge.
  4. Confirmed Remove calls `forget(record.id)`, clears text/source evidence,
     produces a tombstone, and updates linked new queue history to `undone`.
  5. Cancel leaves both stores and UI unchanged; persistence failure is visible
     and retryable.
  6. Legacy records without linkage remain exactly removable; old queue history
     is left unchanged.

## Implementation and initial validation

- Added `originReviewQueueItemId` to confirmed records and preserved it in
  text-free tombstones. Confirmation now supplies the exact queue ID.
- Added `memoryAutoAcceptPaused` with a default/normalizer and a Level 2 Memory
  settings toggle. The trust counter remains monotonic; the runtime gate skips
  auto-confirm only while paused.
- Added record-first Pagelet governance: automatic badge, source-backed record
  card, explicit removal confirmation, exact-ID forget, immediate tombstone UI,
  and durable `applied -> undone` audit reconciliation for linked records.
- Independent review found and the implementation now closes four additional
  trust-boundary cases:
  - automatic confirmation is limited to `low` sensitivity; `medium`, `high`,
    conflicts, and task constraints stay manual;
  - the Memory master setting gates Level 2, so disabling Memory cannot leave a
    hidden automatic confirmation path active;
  - a failed pause-setting save restores the previous UI/runtime value, shows a
    localized error, and allows a later durable retry;
  - a tombstone with an origin link is an idempotent retry marker, reconciled
    immediately, on startup idle, and after settings changes even when Review
    Queue display/production is disabled; confirmation-stage partial success
    converges through `accepted -> applied -> undone`, while an already-applied
    item uses `applied -> undone`;
  - Pagelet detail session cache receives the actual tombstone, preventing
    removed summary/source text from reappearing after native state restore.
- Legacy unlinked records remain exactly removable without queue guessing.
- Before implementation, four suites reproduced eight failing assertions. After
  implementation the combined follow-up bundle passes 7 suites / 339 tests;
  focused type-check, lint, build, whitespace, locale parity, and community DOM
  scan pass.
- Actual scope exceeds eight files for the documented cross-layer reason; no
  dependency, public API, database, authorization, retrieval, or VSS behavior
  changed.
- Post-review focused validation passes 4 suites / 242 tests plus TypeScript,
  lint, and whitespace checks.
