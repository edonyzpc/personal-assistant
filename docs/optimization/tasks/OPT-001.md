# OPT-001 — Preserve historical Memory candidates at Level 2

## Pre-implementation revalidation

- Original problem: entering Level 2 during plugin idle, or manually confirming the threshold item, schedules a scan over every historical `suggested` Memory candidate.
- Root cause: `autoConfirmPendingMemoryCandidates()` is wired both as a backlog sweep and as creation-time behavior, although `createReviewQueueItem()` already owns the new-candidate path.
- Modification boundary: `src/plugin.ts` plus focused plugin regression tests.
- Non-goals: no trust threshold, record format, conflict rule, UI, or confirmation-count change.
- Acceptance:
  1. Level 2 idle does not touch existing candidates.
  2. The manual confirmation that reaches Level 2 does not touch another existing candidate.
  3. A newly created eligible candidate at Level 2 is still auto-confirmed.
