## 一、North Star

### 最终定稿

> **随手记下，需要时自然浮现。**

### 演进过程

| 轮次 | 候选 | 淘汰原因 |
|------|------|---------|
| 1 | 安静且可信 | 描述工具性格而非用户收益，是设计约束不是价值主张 |
| 2 | 让个人知识自然复利 | "复利"是比喻不是定理，学术上缺乏实证（ChatGPT 报告确认） |
| 3 | 你的笔记越写越有用 | 过度承诺 — 不是所有笔记都应该越来越有用 |
| 4 | 随手记下，持续增值 | "增值"接近"复利"，同样缺乏实证支撑 |
| 5 | 随手记，问得到，想得起 | "问得到"(Chat)不是 PA 差异化，与 Copilot 正面竞争 |
| 6 | 让 vault 活起来 | 太抽象 |
| **最终** | **随手记下，需要时自然浮现** | 通过三个测试 ✓ |

### 三项测试验证

1. **用户测试**: 用户能想象场景 — "我在写东西，旧笔记自己出现了" ✓
2. **取舍测试**: 新功能是否让"浮现"更准确、更及时？能做判断 ✓
3. **竞品测试**: Copilot 是"你问它答"(主动搜索)，PA 是"它自己出现"(被动浮现) ✓

### North Star 展开版

> PA 让你用最低成本留下想法，然后让值得回来的旧笔记在你需要的时候自然出现 —
> 保持 vault 有序，每次使用都让下一次更好。

## 二、产品哲学（设计约束，不是产品定位）

### 核心哲学

> **安静且可信。**

这是 PA 交付价值时的姿态约束，不是面向用户的价值承诺。

### 与 North Star 的关系

```
North Star（用户价值）  →  随手记下，需要时自然浮现
        ↓ 用什么方式交付
产品哲学（设计约束）    →  安静且可信
        ↓ 具体怎么做
设计原则（实现规范）    →  见下文
```

### 关键认知

**产品哲学 ≠ 产品价值。** 哲学指导内部设计决策，价值面向用户传达。
"安静且可信"告诉开发者怎么做，"随手记下，需要时自然浮现"告诉用户为什么用。

## 三、产品设计原则与约束

### 来自研究报告的原则（两份报告共识）

| # | 原则 | 研究支撑 | PA 含义 |
|---|------|---------|---------|
| 1 | "整理"是表面需求，"用得上"才是真实需求 | Jones PIM 框架、归档者vs堆积者研究 | 但整理也服务情绪需求（控制感、安心感），不能完全忽视 |
| 2 | 知识的价值在于被复用，不在于被整齐存放 | 收藏谬误研究、MIT 笔记生命周期研究 | PA 不优化"整齐"，优化"在需要时出现" |
| 3 | AI 应帮用户"想起"而非"替用户思考" | U Michigan 2025：中等AI > 高度自动化 > 无AI | PA 不做内容生成，做上下文触发的知识重现 |
| 4 | 知识复利是比喻不是定理 | ChatGPT 报告：缺乏纵向 PKM 实证 | 不在产品定位中承诺"复利"，用"复用"替代 |
| 5 | 不是所有笔记都值得被拯救 | MIT：28% 笔记在 10 天内被删除 | PA 允许低价值笔记自然衰减 |
| 6 | 最有价值的重现是零查询的上下文触发 | JITIR 研究、Context-Triggered Recall | Quiet Recall 的核心设计方向 |
| 7 | 用户欢迎主动帮助但抵制自主控制 | 78% 欢迎主动帮助，仅 12% 接受完全自主 | PA 可以主动推送但不能自主执行 |

### 保留的设计约束（从原 North Star 降级）

| 约束 | 内容 | 审查标准 |
|------|------|---------|
| Less management, more capture | 降低捕获摩擦，不增加管理负担 | 用户能否忽略这个功能而不产生惩罚？ |
| Less generation, more return | 不生成新内容，重现用户自己的旧内容 | 展示的是用户写的还是 AI 写的？ |
| Less interruption, more right-time presence | 在自然断点推送，不在编辑过程中打断 | 触发时机是用户的注意力间隙吗？ |
| Less black-box, more source-backed | 每个推荐都有来源和 whyNow | 用户能追溯到原始笔记吗？ |
| Less automation, more earned trust | 从手动确认到范围授权逐步升级 | 这个自动化程度是用户授权的吗？ |

### 衰减原则（新增）

**不做硬衰减，用多维信号排序替代。**

| 维度 | 作用 |
|------|------|
| 语义相关性 | 主信号 — 决定"该不该出现" |
| 时间新鲜度 | 辅助信号 — 同等相关时优先新的 |
| 连接密度 | 质量信号 — 被引用多 = 可能更重要 |
| 笔记丰富度 | 质量信号 — 短临时备忘 vs 深度思考 |
| 用户反馈 | 学习信号 — 被反复忽略 = 降权 |

原因：硬衰减（时间乘数 + 覆写阈值）在最需要衰减的场景中失效 —
同主题大量旧笔记的语义相关性会批量触发覆写，导致衰减形同虚设。

## 四、产品架构重新认识

### PA 的两个用户面 + 一个后台

```
用户看到的：
┌──────────┐     ┌──────────┐
│   Chat   │     │ Pagelet  │
│  拉取式   │     │  推送式   │
│ 用户问    │     │ PA 准备   │
│ PA 答     │     │ 用户选看  │
└────┬─────┘     └────┬─────┘
     │                │
     └───────┬────────┘
             │
用户看不到的：
┌────────────────────────┐
│      PA 智能层          │
│  索引 · 提取 · 检测     │
│  准备 · 评估 · 维护扫描  │
└────────────────────────┘
```

**关键认知**：之前错误地把"自动化"当作第三个用户面。
实际上自动化是 Chat 和 Pagelet 共享的后台智能层，不是独立的产品表面。

### PA vs Copilot 的差异化

| | Copilot | PA |
|---|---------|-----|
| 核心隐喻 | AI 对话伙伴 | 知识重现引擎 |
| 主交互 | "跟笔记聊天" | "旧笔记自然浮现" |
| 内容生成 | ✅ 摘要、改写 | ❌ 不做 |
| Vault 整理 | ❌ | ✅ Maintenance Review |
| 信任层记忆 | ❌ | ✅ Evidence → Memory |
| 文字选择操作 | ✅ Quick Commands | ⚠️ 需要补充基础支持 |

共存策略：**不需要特别处理。** PA 做好差异化功能，Chat 作为基础能力保留但不主推。

## 五、产品功能讨论结论

### 1. Quiet Recall — 核心功能重新定位

**候选池扩展**：从仅 Saved Insights 扩展到**整个 vault**。

原因：
- 解决冷启动死锁（新用户无 Saved Insights 时 Recall 无法工作）
- 与研究报告 #1 机会对齐（vault 级上下文触发重现）
- 不依赖用户先积累 insights

**与 Smart Connections 的区分**：

| | Smart Connections | PA Quiet Recall |
|---|---|---|
| 展示 | 侧栏列表，始终可见 | 1-3 条精选，特定时刻浮现 |
| 内容 | "这些笔记相似" | "你之前写过这个，**因为这个原因**现在可能有用" |
| 解释性 | 无 | 有 whyNow + sourceRefs |

**触发模型**：三层触发

| 触发方式 | 时机 | 性质 |
|---------|------|------|
| 打开/切换笔记 | 自然间隙 | 被动 — PA 自动 |
| 保存后 | 自然断点 | 被动 — PA 自动 |
| 快捷键（如双击 Ctrl） | 用户需要时 | 主动 — 用户发起 |

不做编辑时实时触发（违背"记录时不表现智能"，且 Smart Connections 已占该位置）。

**当前实现 vs 目标设计的差距**：

| 方面 | 当前实现 | 目标设计 |
|------|---------|---------|
| 候选池 | `QuietRecallBuildInput.savedInsights`（仅 Saved Insights） | 整个 vault 的笔记 |
| 触发 | 打开笔记时 | 打开笔记 + 保存后 + 快捷键 |
| 排序 | 固定的 relation scoring（current/related/far） | 多维信号排序（语义 + 时间 + 连接 + 丰富度 + 反馈） |
| 建链能力 | 无（只能保存为 insight 或加入 review queue） | 一键在 frontmatter 添加 `pa-related` |

实现优先级：先扩展候选池到 vault，再补触发模型，最后加建链能力。

### 2. Weekly Review — 拆解

**结论：Weekly Review 作为独立功能拆解。**

**完整论证（供 Agent 理解决策依据）**：

1. **习惯依赖问题**：Weekly Review 要求用户每周主动触发。MIT 研究显示 75% 的
   笔记从未被再次编辑，PKM 系统失败的首要原因是用户无法维持高强度维护节奏。
   做 weekly review 的人是 Obsidian 用户中的极小比例（GTD/Bullet Journal 实践者）。

2. **功能重叠**：如果 Quiet Recall 扩展到 vault 级别并在日常写作中持续重现旧笔记，
   Weekly Review 中的"Recall 候选"和"本周笔记"section 变成重复。

3. **负担问题**：五个 section（本周笔记、洞察、记忆候选、维护提案、Recall 候选）
   一次性呈现 = "PA 这周积攒的所有东西一次性倒给你"，违背低负担原则。

4. **为什么不保留为可选功能**：即使可选，它的存在暗示"你应该每周回顾"。
   研究报告明确警告：间隔重复硬套到创意型知识管理中会把开放式探索变成
   有压迫感的"审查债务"。不如把有价值的部分融入日常流。

**有价值的内核重新归位**：

| 内核 | 归入 | 与原 Weekly Review 的区别 |
|------|------|--------------------------|
| Memory 候选批量确认 | Pagelet Tab 常驻 section，不绑定周频 | 用户任何时候想确认都行，不需要等"周末" |
| 跨笔记模式检测（主题、张力） | PA 积累足够材料时主动 nudge | PA 决定什么时候有值得展示的模式，不是固定周频 |

### 3. Maintenance Review — 定位确认

**最具体、最独特、最容易理解的 PA 价值点。**

- 没有竞品做 vault 整理提案
- 用户痛点真实（"inbox 堆了 50 篇没整理"）
- 理解成本极低（"帮我整理笔记" 谁都懂）
- 但"整理"是手段不是目的 — 目的是让 vault 保持可导航、可检索

补充认知：整理同时服务**功能需求**（检索效率）和**情绪需求**（控制感、安心感）。

### 4. 建立连接 — 通过 Frontmatter

**方案**：用户在 Quiet Recall 中点击"关联到当前笔记"后，PA 在 frontmatter
中添加 `pa-related` 条目。

```yaml
---
pa-related:
  - "[[旧笔记标题]]"
---
```

**已验证**：Obsidian v1.4+ 原生支持 frontmatter 中的 `[[wikilink]]`：
- Properties 视图渲染为可点击链接 ✅
- Graph View 显示关联关系 ✅
- Backlinks 面板显示反向引用 ✅
- resolvedLinks 自动包含 ✅
- 笔记正文不变 ✅

**设计优势**：PA 只需在 frontmatter 加一行，Obsidian 原生能力负责其余一切。

### 5. Quick Capture — 定位确认

**即时价值入口，但不是核心差异化。**

- 3 秒留下一个想法（即时价值）
- 但 Obsidian 已有 QuickAdd/Daily Notes（增量优化，非独特价值）
- 真正的价值是捕获的内容进入 PA 的重现循环

### 6. Chat — 基础能力，不主推

- 保留但不作为核心卖点
- 不与 Copilot 在 Chat 赛道竞争
- 需要补充基础的文字选择操作支持
- 对没有安装 Copilot 的用户提供完整体验

### 7. Graph Discovery — 连接驱动复利

研究结论：驱动知识复利的关键行为是"建立连接"，不是"写更多"也不是"回顾更多"。
PA 通过 Quiet Recall 发现关联 + Frontmatter 一键建链来降低建立连接的摩擦。

## 六、首次体验设计

### 一次性桥梁引导方案

| 触发条件 | 引导内容 | 展示方式 | 次数 |
|---------|---------|---------|------|
| 首次安装 + vault > 50 篇笔记 | "PA 可以帮你扫描笔记并建议整理" → Maintenance Review | **PA 统一通知样式**（Bubble/Pet） | 1 次 |
| 首次 Quick Capture 完成 | "你的想法已保存。PA 会在你未来写相关内容时提醒你" | PA 统一通知样式 | 1 次 |
| 首次 Quiet Recall 浮现 | "这是 PA 发现的与当前笔记相关的旧笔记" | PA 统一通知样式 | 1 次 |

**约束：所有引导通知使用 PA 统一的通知样式（Bubble/Pet 状态变化），
不使用 Obsidian 原生 Notice**。原因：原生 Notice 可读性低且容易被忽略。

### 不同用户的最快价值入口

| 用户类型 | 最快入口 | 时间 |
|---------|---------|------|
| 老 vault（100+ 笔记）+ 好奇型 | Chat："帮我找找以前写过关于 X 的" | 30 秒 |
| 老 vault + 整理型 | Maintenance Review | 5 分钟 |
| 新 vault | Quick Capture 积累材料 | 延迟 1-2 周 |

## 七、竞品定位

```
Copilot：跟你的笔记聊天（对话式 AI 助手）
Smart Connections：找相关笔记（语义相似度雷达）
PA：让需要的旧笔记在需要时自然浮现（知识重现 + vault 照料）
```

PA 与 Copilot 共存策略：**不需要特别处理**，各做差异化。

## 八、研究报告关键发现摘要

### 两份报告共识

1. 绝大多数笔记写后就死了（75% 从未编辑，仅 34% 被有意重找）
2. 上下文触发的重现是最高价值的知识管理能力
3. AI 应辅助而非替代思考（中等 AI > 高度自动化）
4. 用户要主动帮助但不要自主控制（78% vs 12%）
5. PKM 系统失败的核心是 capture 到 retrieval 之间的上下文断裂

### ChatGPT 对 Gemini 的三个修正

1. "整理是表面需求"方向对但太绝对 — 整理也服务情绪需求
2. 不是所有笔记都值得被拯救 — 有些就是一次性的
3. "知识复利"是比喻不是定理 — 用"复用"替代

### 产品战略指向

> "The highest-value opportunity is to reduce the cost of turning captured
> material into timely, trustworthy re-use."
>
> 最高价值的机会是降低"把已捕获的材料转化为及时、可信的复用"的成本。
