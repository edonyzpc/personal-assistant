架构

严重 plugin.ts 已成为上帝对象（2119 行，~28 个 Pagelet 方法） 架构

问题描述

src/plugin.ts 的 PluginManager 类已膨胀至 2119 行，本次新增约 28 个 Pagelet 相关方法。其中 runPageletReviewForFiles 单个方法就有 260 行（L742-L1001），包含了完整的审阅编排链路：

  • LLM 审阅编排（构建 PageletReviewModel，调用 reviewNote）
  • 作用域管理（scope plan 构建、文件解析）
  • UI 协调（pageletView.showReviewStarted/showReviewProgress/showReviewResult）
  • 写操作编排（runtime.actionExecutor.execute）
  • 取消/中止处理，包含嵌套的 if (abortController.signal.aborted) 检查（L839、L844、L889、L906）
  • 计时遥测

这违反了单一职责原则。随着 Pagelet 持续演进（特别是 Write Action Framework v1 的集成），该文件会继续膨胀，测试和维护成本将急剧上升。

建议

将 L742-L1001 的编排逻辑提取到独立的 PageletReviewOrchestrator 类中（src/pagelet/pagelet-review-orchestrator.ts）。plugin.ts 只负责生命周期管理和模块连接。编排器通过依赖注入接收 PluginManager。

严重 PageletView ↔ PluginManager 双向耦合 架构

问题描述

src/pagelet/view.ts:117 注入了 PluginManager，在 L314 直接调用 this.plugin.runPageletReviewForPageletScope()，L374 调用 this.plugin.refreshPageletScope()，L595/L606/L614 调用多个 plugin 方法。同时 plugin.ts 也引入并操作 PageletView — 形成事实上的双向依赖。

虽然 TypeScript 不会报错（view 使用 import type），但这种紧耦合导致 PageletView 无法独立测试，任一方向的接口变更都会级联。

建议

PageletView 应通过事件/回调接口与宿主通信。将 L314 等处的直接调用改为构造时注入的回调（如 onReviewRequested: () => void、onRefreshScope: () => void），将 view 对 plugin 的依赖收窄为仅含必要回调的接口。

重要 AgentControlSnapshot 的不可变性假象（ReadonlySet 仅编译时约束） 架构

问题描述

src/ai-services/pa-agent-control-policy.ts — 所有 derive* 函数（L178、L209、L239）使用展开运算符拷贝 snapshot。然而 ReadonlySet 仅是编译时约束，如果调用者在创建 snapshot 后修改了底层 Set，变更会泄漏到 snapshot 内部。

当前没有调用者这样做，但这是一个脆弱的不变量，一旦违反会静默破坏。

建议

在 createAgentControlSnapshot 中统一深拷贝所有集合字段。diagnostics 数组也应深拷贝（当前 L98 只做浅拷贝，每个 diagnostic 的 metadata 可能被共享）。

重要 PaAgentLoop 承担了过多底层关注点（1671 行） 架构

问题描述

src/ai-services/pa-agent-loop.ts 同时处理：流式 chunk 分类（L320-474）、缓冲 tool call 去重（L596-609、L732-807）、工具执行超时/中止/宽限期管理（L836-918、L956-989）、并行/串行/混合执行策略（L596-730）、observation budget（L1094-1103）、turn 计时遥测（L553-578）。

runTurn 方法单独就有 274 行（L320-594）。每个关注点单独实现质量不错，但组合在一起使得维护成本很高。

建议

考虑拆分出 ToolExecutionDispatcher（管理并行/串行/混合策略）和 ModelChunkConsumer（管理流式 chunk 解析和分类）。当前不阻塞，但应在下一个重构迭代中处理。

重要 runPageletReviewForFiles 中 abortReason 竞态条件 架构

问题描述

src/plugin.ts:755-759 — abortReason 赋值和 abortController.abort() 不是原子操作。两个事件监听器（L794 active-leaf-change 和 L795 file-open）可能并发触发。

更关键的是，L889/L906 的 if (abortReason) 检查与 L879 的 LLM await 之间存在 TOCTOU 窗口 — 如果中止恰好发生在 reviewModel.reviewNote 返回之后但在条件检查之前，结果会被丢弃但用户看到的是 "已中止" 而非实际结果。

建议

统一使用 abortController.signal.aborted 进行状态检查，而非单独的 abortReason 变量。通过 AbortController.abort(reason) 附加原因，从 signal.reason 读取。

一般 scope.ts 日期范围使用毫秒运算（夏令时风险） 架构

问题描述

src/pagelet/scope.ts:395-396 使用 todayStart - (days - 1) * 86400000 计算日期范围。在有夏令时的时区，一天并不总是 86,400,000 毫秒。"最近 7 天" 范围在夏令时切换周内可能错误分类边界笔记。

建议

使用 new Date(year, month, date - days) 构造器代替毫秒减法。Date 构造器会自动处理月份/日期翻转和夏令时。

一般 PageletView 持有 14 个可空 DOM 字段 架构

问题描述

src/pagelet/view.ts:80-107 声明了 14 个 HTMLElement | null 字段。onClose（L148-166）逐个置 null。新增 UI 元素时容易遗漏置 null，导致内存泄漏。

建议

将所有 DOM 引用统一到一个 ViewState 对象中。关闭时只需 this.state = null 即可释放全部引用。

严重 审阅启动前未展示 token/费用预估 产品

问题描述

产品设计文档（docs/review-assistant-product-design.md:447-449）明确要求 "Show candidate count and estimated scale before running"。当前实现只在 scope 区域展示文件数量和路径，没有预估 token 用量或费用范围。

唯一的费用相关文案是 src/locales/pagelet/en.json:116 的 reviewSelectedDescription："Selected note text may be sent to your configured AI provider and may use credits" — 但这只是 tooltip/aria-label，不是一个显式确认步骤。

对于使用付费 API 的用户，这是信任和成本控制的关键缺失。

建议

在点击 "审阅选中" 后增加一个轻量确认步骤，展示预估 token 数量和费用范围。或者在 scope 区域中文件数量旁显示预估 token。

严重 取消审阅后 scope 状态未刷新 产品

问题描述

src/pagelet/view.ts:251-257 — 中止后状态设为 idle 并显示 "审阅已停止"，但 scope plan 未重新渲染。src/plugin.ts:818-819 只调用了 pageletView?.showReviewAborted(file.path)，未刷新 scope。

如果用户在 "最近 7 天" 范围下取消了长审阅想重试，scope 可能已过时（审阅期间可能有新文件创建）。产品设计文档（L914-919）要求 "Preserve selected range and included/skipped state"。

建议

中止后调用 refreshPageletScope 重新加载当前 scope，确保用户重试时笔记列表与实际状态一致。

重要 审阅进度缺少量化指标 产品

问题描述

src/locales/pagelet/en.json:122-136 定义了丰富的状态文案（reading、preparing、checkingLimits 等），在 src/plugin.ts:1017-1036 formatPageletReviewProgress 中使用。但这些都是纯文本状态 — 没有百分比、预估剩余时间或步骤编号。

审阅超时设为 PAGELET_REVIEW_TIMEOUT_MS = 90,000，可能持续数十秒。用户只看到文字在变化但不知道要等多久，移动端尤为突出。

建议

在进度面板添加步骤指示器（如 "步骤 3/6"）。超过 10 秒后显示预估时间如 "通常需要 20-60 秒"。

代码质量

严重 createToolInterruptPromise 定时器泄漏风险 代码

问题描述

src/ai-services/pa-agent-loop.ts:997-1036 — 工具正常完成时，finish() 设置 settled=true 防止重复 resolve，但 toolTimeoutTimer 和 wallClockTimer 仍在内存中运行，直到 finally 块调用 cleanup()。

如果 finally 块在到达 cleanup() 前抛异常，定时器永远不会被清理。在高频工具调用场景下，泄漏的定时器会累积。

建议

在 finish() 函数内部也调用 cleanup()，确保每条 resolve 路径都能立即清理定时器。

严重 Workspace 事件监听器未通过 Obsidian 生命周期注册 代码

问题描述

src/plugin.ts:794-795 通过 this.app.workspace.on() 注册 active-leaf-change 和 file-open 监听器，在 finally 中手动 offref()。未使用 this.registerEvent()。

如果审阅期间插件被卸载（用户禁用插件），这些监听器会永久泄漏，因为异步流的 finally 块可能不会执行。

建议

使用 this.registerEvent() 实现插件卸载时的自动清理，同时在 finally 中保留显式 offref 用于正常流程清理。

重要 不安全的类型断言：as WriteActionCapability 代码

src/ai-services/pa-agent-runtime.ts:605 — 仅检查 capability.kind !== "action" 后就将 capability 断言为 WriteActionCapability。如果 kind === "action" 但对象缺少 executeWrite 方法，运行时会崩溃。

建议

增加运行时类型守卫：在使用前检查 if (typeof writeCapability.executeWrite !== 'function')。

重要 Buffer.byteLength 在移动端（iOS/Android WebView）可能不存在 代码

src/pagelet/pa-review-tool-provider.ts:351 使用 Buffer.byteLength(body, "utf8")。Obsidian 桌面端基于 Electron 有 Node.js Buffer，但移动端是纯浏览器 WebView，Buffer 可能不存在。该 provider 声明 platform: "both"。

建议

替换为 new TextEncoder().encode(body).length 以实现跨平台兼容。

重要 MutationObserver resize 防抖定时器可能在插件卸载后触发 代码

src/plugin.ts:228-248 — setTimeout 回调可能在已卸载的插件实例上调用 this.localGraph.resize()。

建议

在 setTimeout 回调中增加活跃性检查，例如 if (!this._loaded) return;

一般 truncateOverlongFields 直接原地修改输入对象 代码

src/pagelet/pa-review-schemas.ts:523-574 尽管声称 "Returns the mutated payload"，但直接修改了输入对象。持有原始对象引用的调用者会看到意外变更。

使用浅拷贝 { ...root }，或在文档中明确说明这是原地修改。

一般 diagnostics.timings 每次追加都创建新数组 代码

src/pagelet/pa-review-model.ts:1130 — diagnostics.timings = [...(diagnostics.timings ?? []), entry] 每次 recordPageletTiming 调用都创建新数组。在最多 4 次 LLM 重试的场景下，产生不必要的 GC 压力。

一般 View DOM 事件监听器未通过 Obsidian API 注册 代码

src/pagelet/view.ts:312-313 使用原生 addEventListener 为按钮绑定点击事件，未使用 this.registerDomEvent()。如果外部引用持有按钮元素，监听器在 view 关闭时不会被移除。

亮点（18 项）

架构
纯函数式控制策略状态机
pa-agent-control-policy.ts 采用纯函数 + 不可变数据结构设计。所有 create*/derive* 函数返回新 snapshot 而不修改输入。Exposure mode 转换路径（semantic-first → source-scoped → answer-ready → final-only）清晰可追溯。
架构
scope.ts：零 Obsidian API 依赖的纯数据模块
scope 模块不依赖 Obsidian API 或 plugin 层（仅引入 normalizePath）。完全通过输入数据（PageletScopeFileLike）驱动，高度可测试 — 246 行 scope 测试验证了这一点。
架构
PageletReviewModel 8 行失败矩阵 + 级联回退
覆盖 schema 不匹配、缺失 source_id、空建议、截断、部分解析、超时和解析错误。两条路径级联（structured → JSON mode），带有纠正重试和全程诊断/计时遥测。
架构
混合工具调度："任一串行则全批串行"
resolveBatchExecutionMode 实现了优雅的调度策略：默认并行提升吞吐量，任何声明 executionMode: "sequential" 的工具会将整个批次降级为串行。简单、安全，无需复杂的依赖图。
架构
预飞检查的防御性分层
classifyPreflightSkip（L732-807）按精心设计的优先级顺序检查：final_answer_only 最先，然后是控制快照检查、占位符检测、预算检查、schema 验证、去重检测。注释说明了优先级理由。
代码
XSS 安全的 SuggestionCard 渲染
使用纯数据描述符模式（markup builder）+ DOM API（setText/setAttribute），完全避免 innerHTML。所有用户生成内容通过 setText()（等同于 textContent）设置，天然防止 XSS。
代码
工具观测沙箱化防 prompt injection
formatToolObservations 将工具输出包裹在 <untrusted> 标签中，通过 escapeUntrustedBoundary 防止标签注入。escapeAttributeValue 对属性值中的 "<>& 进行消毒。有效的 prompt injection 防御层。
代码
normalizeReviewsFolder：10 种拒绝原因的深度安全
Unicode 不可见字符检测、NTFS 尾部点/空格剥离、大小写折叠后的 .obsidian 检查、父目录遍历检测。每种拒绝原因映射到 i18n key，通过 PageletReviewsFolderError 类型确保编译期穷举检查。
代码
mintNonCollidingPath：优雅的碰撞降级
先尝试 -2 到 -100 后缀，再降级到 HHMMSS 时间戳后缀。确保在极端场景下文件写入也不会被阻塞。markSelfWrite 在实际写入前调用，正确实现 Write Action Framework 的重入防护。

样式
全面使用 Obsidian CSS 变量
新增约 750 行样式几乎全部使用 Obsidian 原生变量：--background-primary、--text-normal、--interactive-accent、--font-ui-small、--shadow-s 等。确保与第三方 Obsidian 主题的良好兼容性。
样式
一致的 BEM 命名 + pa-pagelet- 前缀
所有新增类严格遵循 pa-pagelet-<block>__<element>--<modifier> 模式。pa-pagelet- 前缀有效防止了与 Obsidian 内部或其他插件的样式冲突。
用户体验
Scope 透明性：用户精确控制发送给 AI 的数据
Scope 机制（包含/跳过列表、逐条 checkbox、排除原因标签、聚合审阅输出笔记数）在 AI 笔记插件中提供了少见的透明度。用户可以精确控制和了解哪些数据被发送给 AI provider — 这是重要的信任建设。
用户体验
扎实的无障碍基础设施
aria-live 区域、prefers-reduced-motion 全局动画禁用、Mod+/ 焦点跳转命令、包含上下文信息的按钮 aria-label、屏幕阅读器播报（pagelet.a11y.announce.done/error）。超出大多数 Obsidian 插件的无障碍基线。
用户体验
"审阅只写独立笔记，绝不修改源笔记" 的产品边界
从产品设计文档到代码实现（Write Action Framework + self-write 重入防护），这条安全边界被严格执行。这种克制在 AI 工具中难得一见，保护了用户数据完整性。
