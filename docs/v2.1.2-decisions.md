## 0. 决策驱动原则

- **bundle size 不作为产品决策驱动力** —— Obsidian 用户对插件几 MB 差距无感知；架构精简不应以 bundle 为衡量
- **优先用户痛点** —— 复议触发条件统一改为用户感知（移动端冷启动 / OOM / 配置丢失等),不再用代理指标(组件数量、bundle 体积)
- **里程碑 + 时间下限** —— v2.x 不绑日历日期,按"功能就绪即发";时间窗仅作硬下限(如 v2.5 ≥ 2026-11-29)
- **Plan B 不写代码** —— 备用方案保留为决策记录,不预先投资实现;触发条件满足后再开 SDD

## 2. 八项不确定性讨论拍板

### Q1. @sqliteai 脱钩时机:v2.3 出版本(机会 spike)

**决策**: B' — v2.3 出版本,v2.2 期间允许"机会 spike"

**Why**:
- v2.2 已塞满 P0 + flag 清扫 + Plan C 拆 SDD,再加 spike 容易过载
- @sqliteai 当前还能用,不是"今天着火"级别
- spike 跟迁移合并比拆到两个 minor 中间空 2 个月效率高
- E2 月度回归在 v2.2 期间能给"上游 fork 是否复活"的信号

**How to apply**:
- v2.2: CI 加月度 spike `npm install @sqliteai/sqlite-wasm@latest` 全量回归(E2 策略)
- v2.2: 如 P0 提前完成可启动机会 spike,**但不作为 v2.2 发版条件**
- v2.3: 第 1 周 spike 验证 + 第 2 周迁移 + 第 3 周回归

实施细节见 [`sdd-sqliteai-supplier-migration.md`](./sdd-sqliteai-supplier-migration.md)。

### Q2. keychain 迁移:v2.5 删迁移代码(迁移逻辑已就绪)

**决策**: A — v2.5 PR 内删除 ~110 行迁移代码(`./src/settings.ts:59` / `./src/utils.ts:189-192` / `./src/plugin.ts:117,1172-1196,1219-1221`)

**Why**:
- 迁移代码已在 v2.0(commit `04a16d4`,2026-05-29)引入并自动跑——**不需要"keychain 迁移工具"**
- 之前 memory 写的 "v2.4 出 keychain 迁移工具" 是误读,已修正
- `@deprecated Remove after v2.5.0` 注释已写在代码里——是历史承诺,无理由跨期
- v2.0 → v2.4 共 ≥ 5 minor 迁移窗口,对 Obsidian 插件用户已经足够

**How to apply**: v2.4 解放出来,可做 Action mode 早期 / Skill 扩展 / 其他 P1。

实施细节见 [`sdd-apitoken-cleanup.md`](./sdd-apitoken-cleanup.md)。

### Q3. P0 8 项内部顺序:批 1 (#1+#2+#4) 同 PR + 批 2 (#5/#6/#7/#8) 并行

**决策**: A — 批 1 同 PR 串行 commit;批 2 各独立 PR 并行;#3 推迟到 6/12 后

**Why**:
- #1+#2+#4 改同一组用户体验链路,做一个 PR 保持文案/截图/操作一致
- #5/#6/#7/#8 单文件改动 + 测试不相关,并行 PR 不冲突
- #3 实际是日期绑定 deprecated("Will be removed after 2026-06-12"),按 [[deprecated-removal-convention]] 推迟

**How to apply**:
- 批 1 同 PR commit 顺序:#1 (ribbon) → #2 (空状态文案) → #4 (README + Manual)
- 批 2 PR 顺序按文件改动从小到大:#7 → #5 → #6 → #8
- #3 在 6/12 之后第一次 PR 处理(v2.2 窗口)

实施细节见 [`sdd-chat-onboarding-flow.md`](./archive/sdd-chat-onboarding-flow.md)、[`sdd-command-palette-cleanup.md`](./sdd-command-palette-cleanup.md)、[`sdd-dependency-pruning.md`](./sdd-dependency-pruning.md)。

### Q4. capability kinds:维持 B3 三抽象(action mode 推迟超 2027 H2 则复议)

**决策**: A — 维持 `tool` / `context` / `action` 三抽象 + 加复议触发

**Why**:
- 真实代码冗余仅 7 行(不是想象中的"庞大类型系统")
- `./src/ai-services/policy-engine.ts:35` 的 `kind === "action"` 是未来防御线(v1 阶段任何 action capability 自动拒绝)
- B 合并后 action mode 落地要做 5 处加回,反而花更多工时

**复议触发条件**: action mode 推迟到 **2027 H2 之后**仍未启动,则回看是否合并 `context+action` 为单一 `"future"` 字面量。

实施细节见 [`sdd-tool-registry-collapse.md`](./archive/sdd-tool-registry-collapse.md) 与 [[action-mode-roadmap]]。

### Q5. React 触发条件:去掉"组件数 ≥ 5"(按 §8 视角修正)

**决策**: B 修正版 — 仅留 React 独占特性 + 不兼容第三方库 两条触发器

**Why**:
- 之前"组件数 ≥ 5"基于 bundle 摊薄逻辑——bundle 不是产品驱动力(§8 修正)
- 真正决定"是否要切 Preact"的是"功能可行性",不是"组件数量"

**复议触发条件**:
1. 任意新组件用 React 独占特性(Suspense + lazy / useTransition / useDeferredValue / concurrent / Server Components)
2. 引入 preact/compat 不完全兼容的第三方 React 库

实施细节见 [`sdd-react-preact-evaluation.md`](./sdd-react-preact-evaluation.md) 与 [[react-evaluation-trigger]]。

### Q6. WASM 不动决策的复议时点:用户感知触发,不主动复议

**决策**: 用户视角重设 — 不主动测 bundle,仅用户感知触发

**Why**: bundle size 不是用户感知指标,不应作为复议触发器。

**复议触发条件(任一)**:
1. 移动端用户报告冷启动 ≥ 5 秒(issue / TG / Discord)
2. 移动端 OOM 崩溃 issue ≥ 3 个独立用户
3. 加载阶段被动遥测 P95 ≥ 5 秒

**不触发**: bundle 体积变化(即使 v2.3 后变大或变小都不复议)、桌面端启动时间。

### Q7. 版本时间窗:里程碑 + 时间下限

**决策**: B — 不绑日历日期,按"功能就绪即发"

**Why**: 独立开发者,节奏由"功能就绪"驱动,不是日历驱动。

**v2.x 主线规划**:

| 版本 | 触发条件 |
|---|---|
| v2.2 | "P0 + flag 清扫 + Plan C 拆 SDD" 完成 |
| v2.3 | "@sqliteai 脱钩 + spike + 真机回归" 完成 |
| v2.4 | 解放(跳过 / Action mode 早期 / Skill 扩展 / 其他 P1) |
| v2.5 | 同时满足:≥ 5 minor + 距 v2.0 (2026-05-29) ≥ 6 个月(≥ 2026-11-29) |

**预期 v2.5 落点**: 2026-12 ~ 2027-Q1。详见 [[v2-release-schedule]]。

### Q8. @deprecated 删除口径:3 种风格语义

**决策**: A — 写 [[deprecated-removal-convention]],明确语义

**Why**: 项目已有两种风格在用(版本绑定 / 日期绑定 / 无明确)。`Will be removed after YYYY-MM-DD` 是**硬下限**,不是"建议日期"。

详见 [[deprecated-removal-convention]] memory。
